Solid Design Principles

Design patterns are common architectural approaches. They are universally relevant. The SOLID design principles are a set of five principles aimed at improving software design and making it more understandable, flexible, and maintainable. They are frequently mentioned along design patterns but they are not a part of that group. The acronym SOLID stands for:

  • S - Single Responsibility Principle (SRP) / Separation Of Concerns (SOC)
  • O - Open/Closed Principle (OCP)
  • L - Liskov Substitution Principle (LSP)
  • I - Interface Segregation Principle (ISP)
  • D - Dependency Inversion Principle (DIP)

Single Responsibility Principle / Separation Of Concerns

A class should have only one, primary responsibility, and it musn't have more. It should not be overloaded with "things" it can do. Different classes should handle different tasks, so that they can be, e.g., changed independently.

A God object is a class that has too many responsibilities, and it is considered an antipattern.

Issue


class Book:
    def __init__(self, title, author):
        self.title = title
        self.author = author

    def print_details(self):
        print(f"Title: {self.title}, Author: {self.author}")

    def save_to_file(self, filename):
        with open(filename, 'w') as file:
            file.write(f"Title: {self.title}, Author: {self.author}")

book = Book("1984", "George Orwell")
book.print_details()  # Printing book details
book.save_to_file("book.txt")  # Saving book to a file

The Book class has multiple responsibilities: managing book data, printing details, and saving data to a file. For example, the function that handles saving to a file should be in another class so that it can be modified once for objects using it when, e.g., switching to a database. Each concern should be separated into a different class.

Solution


class Book:
    def __init__(self, title, author):
        self.title = title
        self.author = author

class BookPrinter:
    @staticmethod
    def print_details(book):
        print(f"Title: {book.title}, Author: {book.author}")

class BookSaver:
    @staticmethod
    def save_to_file(book, filename):
        with open(filename, 'w') as file:
            file.write(f"Title: {book.title}, Author: {book.author}")

# Usage for SRP
book = Book("1984", "George Orwell")

# Printing book details
BookPrinter.print_details(book)

# Saving book to a file
BookSaver.save_to_file(book, "book.txt")

Open/Closed Principle

Software entities (classes, modules, functions) should be open for extension but closed for modification, so that we could add a new functionality without changing the existing code. Functionalities should be added throguh extension, not modification.

Issue


class DiscountCalculator:
    def calculate(self, customer_type, price):
        if customer_type == "regular":
            return price * 0.9  # 10% discount
        elif customer_type == "premium":
            return price * 0.8  # 20% discount
        else:
            return price

calculator = DiscountCalculator()
print(calculator.calculate("regular", 100))  # Output: 90.0
print(calculator.calculate("premium", 100))  # Output: 80.0
print(calculator.calculate("new", 100))  # Output: 100.0

Adding a new discount would require modifying the existing class, and modifications can introduce bugs and require retesting the entire method.

Solution


from abc import ABC, abstractmethod

# Base class for discounts
class DiscountStrategy(ABC):
    @abstractmethod
    def apply_discount(self, price):
        pass

# Specific discount strategies
class RegularCustomerDiscount(DiscountStrategy):
    def apply_discount(self, price):
        return price * 0.9  # 10% discount

class PremiumCustomerDiscount(DiscountStrategy):
    def apply_discount(self, price):
        return price * 0.8  # 20% discount

class NoDiscount(DiscountStrategy):
    def apply_discount(self, price):
        return price  # No discount

# DiscountCalculator is now open for extension but closed for modification
class DiscountCalculator:
    def __init__(self, discount_strategy: DiscountStrategy):
        self.discount_strategy = discount_strategy

    def calculate(self, price):
        return self.discount_strategy.apply_discount(price)

# Usage for OCP
# Regular customer discount
regular_discount = DiscountCalculator(RegularCustomerDiscount())
print(regular_discount.calculate(100))  # Output: 90.0

# Premium customer discount
premium_discount = DiscountCalculator(PremiumCustomerDiscount())
print(premium_discount.calculate(100))  # Output: 80.0

# No discount
no_discount = DiscountCalculator(NoDiscount())
print(no_discount.calculate(100))  # Output: 100.0

Liskov Substitution Principle

Objects of a superclass should be replaceable with objects of its subclass without affecting the correctness of the program. We should always be able to substitute a base type for a subtype.

Issue


# Base class for Employee
class Employee:
    def __init__(self, name, salary):
        self._name = name
        self._salary = salary

    @property
    def salary(self):
        return self._salary

    @salary.setter
    def salary(self, value):
        if value < 0:
            raise ValueError("Salary cannot be negative")
        self._salary = value

    def __str__(self):
        return f"Employee: {self._name}, Salary: {self._salary}"


# Subclass for Manager
class Manager(Employee):
    def __init__(self, name, salary, bonus):
        super().__init__(name, salary)
        self._bonus = bonus

    # Overriding the salary setter to include bonus automatically
    @Employee.salary.setter # using the property decorator for x setter from the Employee class
    def salary(self, value):
        if value < 0:
            raise ValueError("Salary cannot be negative")
        # Sets both salary and bonus as a fixed proportion of the salary
        self._salary = value
        self._bonus = value * 0.1  # Bonus is always 10% of the salary

    @property
    def bonus(self):
        return self._bonus


# Client function that expects Employee behavior
def adjust_salary(employee):
    original_salary = employee.salary
    employee.salary = original_salary + 500  # Increment salary by $500
    print(f"Expected new salary: {original_salary + 500}, Actual: {employee.salary}")


# Test cases
employee = Employee("John Doe", 3000)
manager = Manager("Jane Smith", 5000, 500)

print("Using Employee:")
adjust_salary(employee)  # Works as expected

print("\nUsing Manager:")
adjust_salary(manager)  # LSP violation: Manager's behavior is unexpected

Using Employee:
Expected new salary: 3500, Actual: 3500

Using Manager:
Expected new salary: 5500, Actual: 5500.0

Solution


# Base class for Employee
class Employee:
    def __init__(self, name, salary):
        self._name = name
        self._salary = salary

    @property
    def salary(self):
        return self._salary

    @salary.setter
    def salary(self, value):
        if value < 0:
            raise ValueError("Salary cannot be negative")
        self._salary = value

    def __str__(self):
        return f"Employee: {self._name}, Salary: {self._salary}"


# Subclass for Manager (Corrected)
class Manager(Employee):
    def __init__(self, name, salary, bonus):
        super().__init__(name, salary)
        self._bonus = bonus

    # No longer overriding the salary setter to include bonus automatically
    def update_bonus(self):
        self._bonus = self._salary * 0.1

    @property
    def bonus(self):
        return self._bonus


# Client function that expects Employee behavior
def adjust_salary(employee):
    original_salary = employee.salary
    employee.salary = original_salary + 500  # Increment salary by $500
    print(f"Expected new salary: {original_salary + 500}, Actual: {employee.salary}")


# Test cases
employee = Employee("John Doe", 3000)
manager = Manager("Jane Smith", 5000, 500)

print("Using Employee:")
adjust_salary(employee)  # Works as expected

print("\nUsing Manager:")
adjust_salary(manager)  # Works without unexpected side effects

Using Employee:
Expected new salary: 3500, Actual: 3500

Using Manager:
Expected new salary: 5500, Actual: 5500

Interface Segregation Principle

Clients should not be forced to implement interface methods they do not need, meaning we should use smaller, more specific interfaces over larger, more general ones. YAGNI - You Ain't Going To Need It (why force people to implement something the are not going to need)

Issue


from abc import ABC, abstractmethod

# A single interface for all printer functionalities
class Printer(ABC):
    @abstractmethod
    def print_document(self, document):
        pass

    @abstractmethod
    def scan_document(self, document):
        pass

    @abstractmethod
    def fax_document(self, document):
        pass

# Basic printer that only supports printing
class BasicPrinter(Printer):
    def print_document(self, document):
        print(f"Printing document: {document}")

    def scan_document(self, document):
        raise NotImplementedError("BasicPrinter does not support scanning.")

    def fax_document(self, document):
        raise NotImplementedError("BasicPrinter does not support faxing.")

# Usage
basic_printer = BasicPrinter()
basic_printer.print_document("Report.pdf")  # Works
basic_printer.scan_document("Report.pdf")  # Raises NotImplementedError

Why would we even enable using the scan and fax methods for the basic printer if it can't perform these actions.

Solution


from abc import ABC, abstractmethod

# Splitting the interface into smaller, more specific ones
class Printable(ABC):
    @abstractmethod
    def print_document(self, document):
        pass

class Scannable(ABC):
    @abstractmethod
    def scan_document(self, document):
        pass

class Faxable(ABC):
    @abstractmethod
    def fax_document(self, document):
        pass

# Basic printer that only supports printing
class BasicPrinter(Printable):
    def print_document(self, document):
        print(f"Printing document: {document}")

# Advanced printer that supports printing, scanning, and faxing
class AdvancedPrinter(Printable, Scannable, Faxable):
    def print_document(self, document):
        print(f"Printing document: {document}")

    def scan_document(self, document):
        print(f"Scanning document: {document}")

    def fax_document(self, document):
        print(f"Faxing document: {document}")

# Usage
basic_printer = BasicPrinter()
basic_printer.print_document("Report.pdf")  # Works

advanced_printer = AdvancedPrinter()
advanced_printer.print_document("Report.pdf")  # Works
advanced_printer.scan_document("Report.pdf")  # Works
advanced_printer.fax_document("Report.pdf")   # Works

Dependency Inversion Principle

High-level modules should not depend on low-level modules, but on abstractions (abstract classes). Additionally, abstractions should not depend on details, but details should depend on abstractions. We want to depend on interfaces rather than concrete implementations because that way we can swap one for the other.

Issue


class MySQLDatabase:
    def connect(self):
        print("Connecting to MySQL database...")

    def save_data(self, data):
        print(f"Saving {data} to MySQL database.")

class DataHandler:
    def __init__(self):
        self.database = MySQLDatabase()  # Direct dependency on a low-level module

    def save(self, data):
        self.database.connect()
        self.database.save_data(data)

# Usage
data_handler = DataHandler()
data_handler.save("user data")  # Works but tightly coupled to MySQLDatabase

DataHandler depends directly on MySQLDatabase, a low-level module. If the database changes (e.g., to PostgreSQL), DataHandler must be modified, violating DIP and reducing flexibility.

Solution


from abc import ABC, abstractmethod

# Abstraction for a database
class Database(ABC):
    @abstractmethod
    def connect(self):
        pass

    @abstractmethod
    def save_data(self, data):
        pass

# Low-level module: MySQL implementation
class MySQLDatabase(Database):
    def connect(self):
        print("Connecting to MySQL database...")

    def save_data(self, data):
        print(f"Saving {data} to MySQL database.")

# Low-level module: PostgreSQL implementation
class PostgreSQLDatabase(Database):
    def connect(self):
        print("Connecting to PostgreSQL database...")

    def save_data(self, data):
        print(f"Saving {data} to PostgreSQL database.")

# High-level module depends on abstraction
class DataHandler:
    def __init__(self, database: Database):
        self.database = database

    def save(self, data):
        self.database.connect()
        self.database.save_data(data)

# Usage
mysql_db = MySQLDatabase()
postgres_db = PostgreSQLDatabase()

# Injecting dependencies
data_handler_mysql = DataHandler(mysql_db)
data_handler_postgres = DataHandler(postgres_db)

data_handler_mysql.save("user data")       # Works with MySQL
data_handler_postgres.save("user data")    # Works with PostgreSQL
